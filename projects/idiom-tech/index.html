[idiom] 중고거래계의 kream, idiom의 기술을 소개합니다

posted by ynawhocodes,
February 12, 2024

안녕하세요. 오늘은 co-founder로 활동했던 SaSS 서비스, idiom(이디엄)의 개발 이야기를 해보려합니다.

idiom에 대한 간단한 소개를 하자면, 서비스 이름은 idiom(이디엄)으로 간편 수거가 가능한 중고 거래 서비스입니다.
모바일로 수거 신청을 하면, 희망 수거일에 택배사에거 수거하고, 저희가 검수/촬영/업로드해주는 중고 거래 서비스를 창업했습니다. 판매자에겐 판매의 부담을 줄이고, 구매자에겐 정확하고 신뢰있는 정보만을 주고 싶어 해당 아이템으로 창업을 시작했습니다.

평소 내가 만든 제품을 사용자들이 쓰는 걸 좋아하고 창업같은 0에서 1을 만드는 활동을 좋아하기에 졸업 후 창업을 해보고 싶다는 생각에 마음 맞는 사람과 창업을 시작했습니다.

창업 아이템 구축(기획부터 출시까지)에 대한 더욱 자세한 내용은 제품의 오너십을 가지며 개발한 내 첫 SaSS, idiom 해당 페이지에서 확인해주시면 감사하겠습니다.

🏔️ idiom 기술 한눈에 보기

코어: React, TypeScript, Next.js
상태 관리: React-Query
스타일링: TailwindCSS, Framer-motion
패키지 매니저: PNPM, Turpo
DB: Supabase
API handler: Next Route handlers
기타: Github Action, Supabase-Discord Webhook Trigger, Figma

👀 왜 해당 스택을 선택하셨나요?

1) 하나씩 설명

가설을 토대로 만들어진 MVP를 빠른 주기로 검증해야 했기에 빠른 개발 속도와 코드 안정성이 필요했습니다.

  • Next.js

따라서 Next는 SSR, SEO최적화, 페이지 기반 라우팅, 이미지 최적화 관련 기능이 내장돼있어 이러한 요구사항을 간편하게 처리할 수 있을 것 같기 때문에 Next를 선택했습니다.

  • React

재사용 가능한 UI 컴포넌트를 쉽게 작성하기 위한 목적과 가상 DOM으로 빠른 렌더링시키기 위해 React를 사용했습니다.

  • TypeScript

빠른 개발 속도엔 안정성이 중요했기에 런타임 에러를 사전에 방지하기 위해 정적 타입 언어를 사용했고, 정의한 타입 정보를 바탕으로 자동 완성 기능이 제공됨과 가독성/유지보수성을 위해 TypeScript를 사용했습니다.

  • React-Query

서버에서 데이터를 fetching하는 기능이 많이 필요했기에 Recoil보다 API 호출과 캐싱 부분에서 특화된 React-Query를 사용했습니다.

  • TailwindCSS

쉽게 반응형 디자인을 구현할 수 있고 클래스 이름 해당 스타일의 기능이기에 직관적이고 개발 생산성이 좋다고 생각하려 TailwindCSS를 사용했습니다.

  • Framer-motion

React내에서 애니메이션 효과를 구현하기에 가장 간편하다고 생각하여 Framer-motion을 사용했습니다.
자세한 애니메이션 라이브러리 비교는 참고해주시면 감사하겠습니다.

  • PNPM,Turbo

레포 간에 재사용할 리소스가 많이 있을 것 생각하여 Vercel에서 만든 Turborepo로 모노 레포로 구성하였고, 전역 캐시를 사용하기에 중복을 최소화하는 장점 때문에 PNPM을 선택했습니다.

  • GitHub Actions, Discord Webhook

개발에 더 몰두할 수 있도록 Github Action을 통해 이슈 생성 시 브랜치 생성 자동화나 Supabase table 변화가 있을 시 trigger를 통해 Discord로 알림이 오는 자동화 작업을 진행했습니다.

2) 디렉토리 살펴보기

이디엄의 모노레포는 다음과 같이 구성돼 있습니다.

📦apps
 ┣ 📦admin
 ┣ 📦landing
 ┣ 📦web
 ┣ 📦packages
 ┗ ...

이디엄은 어드민/랜딩페이지/메인으로 총 3가지 repo로 분류할 것 같아 모노레포로 관리해도 의존성과 복잡한 유지보수 걱정보단 코드 재사용을 통한 장점이 더 클 것 같기 때문이었습니다.

따라서 이디엄은 총 3가지 repo로 구성되어있고,
api, hook, type, ui, util등을 패키지화하여 모듈을 공유하고 있습니다.

 ┣...

 ┣ 📂hook

📐 이디엄의 규칙

제가 디자이너는 아니라 많이 부족하지만요… 개발자 입장에서 어떻게 디자인 해야 좋을지에 가장 집중했습니다

창업 아이템으로 세운 가설을 검증하기 위해 짧은 주기로 MVP를 배포해야했기에 짧은 스프린트 내에 기획/디자인/개발이 모두 가능하려면 효율적으로 작업하는 게 중요한 관건이었습니다. 또한, 개발 생산성뿐만이 아니라 모든 유저에게도 일관적인 사용 경험을 주기 위해 디자인 시스템이 필요하다 여겼습니다. 따라서 이디엄의 디자인 시스템을 다음과 같았습니다.

1. 디자인에 규칙을 두자

  • color, font는 미리 지정해두기 (폰트는 12px ~ 20px, 4px 단위만 사용)
  • 모든 패딩값은 4px 단위로 지정하기
  • Figma 사용 시 토큰 설정 미리 해두기
  • 컴포넌트는 Figma 컴포넌트화 기능 사용해서 수정 반영 싱크 맞추기
  • 사용자가 누를 버튼의 높이는 40px / 60xp 만 사용하기
  • 사용자가 누르지 않는 컴포넌트 크기도 최소 32px로 구성하기
  • 등등

따라서 다음과 같이 규칙을 설정하고 적용했습니다.

이러한 토큰들은 tailwind.configdhk global.css에 작성해두어 개발 시 디자인 시스템 적용하기 편하도록 구성했습니다.

2. 업무 효율을 높이자

  • 중복 작업을 최소화하고 재사용성을 높이기 위해 개발 완료된 컴포넌트나 컴포넌트화한 아이콘들은 컴포넌트 파일명으로 이름을 변경하여 [개발완료] Frame에 이동시켜두기
  • 브랜치 생성과 같은 부가적인 일은 Github action을 통해 자동화 해두었으니 각자 어떤 작업을 하고 있는지 깃허브 이슈 생성 후 작업하기
  • 들어오는 판매 신청 내용을 매시간 어드민을 통해 확인하기 어려우니 trigger를 통해 apply table 변동 시 discord로 알림가도록 자동화

3. 재사용성을 높이기 위해 최대한 고민하자

가설 검증 과정 중인 아이템이라 디자인을 뒤엎은 일이 잦았고, pivoting에도 빠른 대응을 하기 위해 유지보수 용이한 코드 작성하는 것이 저희에겐 가장 우선시하며 개발했습니다.

우선, idiom의 스키마는 위와 같습니다.
kream에 있는 내 물건 판매 기능과, e-commerce에 있는 쇼핑/결제 기능이 모두 있기에 DB 구조를 설계하는데도 간단하진 않았습니다. 하나의 DB에도 많은 컬럼이 있었기에 클라이언트에서 받은 프로퍼티도 많았습니다. 서버에서 받은 데이터들을 전역적으로 관리하고, 사용할 수 있도록 (앞서 말했듯이) 모노레포로 구성하여 공유할 모듈들은 패키지화 하였고,

재사용성이 높으면서 동일한 레이아웃 내에서 하나 이상의 페이지를 동시에 또는 조건부로 렌더링할 modal같은 컴포넌트들은 Parallel Routes으로 분리하여 진행했습니다.
